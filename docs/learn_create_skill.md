# Learn Create Skill

## 目的

这份笔记基于三份材料整理：

- Claude Code 官方文档：Plugin Marketplaces
- Anthropic 博文：Improving skill-creator: Test, measure, and refine Agent Skills
- 本地文章：`/Users/lu/development/okwind/miscellany/outputs/claude-code-skills-experience-sharing.md`

目标不是复述原文，而是提炼出一套适合当前仓库的 skill 创建方法。

## 一句话结论

一个好的 skill，不是“多写一点说明文档”，而是把某类重复任务封装成一个可触发、可渐进加载、可验证、可迭代优化的能力包。

它至少要同时解决四件事：

1. 什么时候触发
2. 触发后加载哪些最关键的上下文
3. 如何借助脚本/模板/资源把任务稳定做对
4. 如何验证它真的比不用 skill 更好

## 一、什么样的 skill 值得做

本地经验文把 skill 大致归为九类，这个分类很实用。对当前仓库最相关的是下面几类：

- 库与 API 参考：把某个 SDK、CLI、内部约定、易错点封装成稳定知识包。
- 数据获取与分析：把查询入口、数据源、固定分析套路、脚本封装起来。
- 业务流程自动化：把“多步、重复、跨工具”的流程变成一个 skill。
- 代码脚手架与模板：生成目录骨架、模板文件、约定配置。
- 代码质量与评审：把审查标准、验证脚本、测试规范固化下来。
- 运维手册与问题排查：把“症状 -> 调查路径 -> 工具 -> 输出报告”的流程固化。

经验上，值得做成 skill 的任务通常有这些特征：

- 高频重复
- 有稳定边界
- 有明显的最佳实践或坑点
- 靠纯提示词不够稳
- 借助脚本或模板后，成功率明显更高

反过来说，如果一个任务只是一次性说明，或者没有明确触发边界，就不适合急着做成 skill。

## 二、好 skill 的核心设计原则

### 1. 不要讲常识，要讲“偏移模型默认行为”的信息

官方经验明确强调，skill 最有价值的部分不是泛泛而谈，而是把模型从“通用默认做法”推向“你真正想要的做法”。

因此 `SKILL.md` 不该堆基础定义，而应优先写：

- 仓库内的特殊约束
- 常见失败模式
- 模型最容易做错的地方
- 你的团队真正接受的做法

### 2. “Gotchas” 是高信号区

经验文最值得吸收的一点是：skill 的高价值信息往往集中在“避坑指南”。

建议每个 skill 都显式维护一个 gotchas 区块，记录：

- 过去真实出现过的失败
- 为什么会失败
- 如何提前规避
- 如何在输出结果里自查

这部分应该随着使用过程不断追加，而不是一次写完。

### 3. 把 skill 当文件夹，不当单 Markdown

三份材料共同指向一个核心：skill 的强大之处，在于它是一个目录结构，能承载渐进式披露。

推荐把内容拆开：

- `SKILL.md`：触发条件、工作原则、主流程
- `references/`：详细 API、契约、字段、例子
- `scripts/`：确定性脚本、采集器、执行器、校验器
- `assets/`：模板、骨架、示例输入输出
- `tests/` 或 eval 数据：验证 skill 的触发与产出质量

这样做的好处是：主 skill 保持短而强，细节按需加载，不把所有内容一次性塞进上下文。

### 4. 不要把 skill 写得过于死板

官方经验提到一个很重要的风险：过度 railroading。

如果 skill 指令太硬，模型会机械执行，不会根据上下文做合理调整。正确做法是：

- 把必须遵守的约束写死
- 把可变策略写成建议或判断条件
- 给出输出标准，而不是过度规定每一句话怎么写

### 5. description 不是简介，是触发规则

Anthropic 对 description 的定位非常关键：它不是给人看的摘要，而是给模型判断“什么时候该触发这个 skill”的入口。

因此 description 应该偏向：

- 任务类型
- 触发关键词
- 适用场景
- 不适用场景

而不是只写“这是一个用于 xxx 的 skill”。

## 三、skill 不能只会写，还要会测

这部分是 2026-03-03 那篇 skill-creator 博文最重要的新变化。

### 1. 要区分“输出质量”与“触发质量”

一个 skill 即使内容很好，如果触发不对，整体效果仍然差。需要分开看：

- 输出质量：skill 被调用后，产出是否更好
- 触发质量：该触发时是否触发，不该触发时是否安静

### 2. 用 eval 验证 skill

官方把 skill 开发往“软件工程化”推进，核心就是：

- 给 skill 写 eval
- 跑 benchmark
- 比较修改前后效果
- 把回归问题尽早发现

这意味着 skill 不应该只靠“我感觉更好了”来迭代，而要积累最小可执行验证集。

对本仓库，推荐至少准备两组样本：

- 正样本：应该触发 skill 的任务描述
- 负样本：不应该触发 skill 的任务描述

再加一组输出验收标准：

- 文件是否生成
- 结构是否正确
- 关键字段是否齐全
- 是否调用了预期脚本

### 3. 随模型演进，skill 会退化

官方博客强调一个现实问题：模型更新后，旧 skill 可能不再表现一致。

因此 skill 不是“写完就结束”，而应进入维护循环：

1. 收集真实使用失败
2. 更新 description 或 gotchas
3. 增补 eval
4. 重新 benchmark
5. 再发布

这也是 skill-creator 新能力的核心价值。

## 四、如何分发与管理 skill

Plugin Marketplaces 文档给出的重点，不只是“怎么上架”，而是“怎么把 skill 当产品分发”。

### 1. 小范围协作可直接进仓库

如果 skill 只服务某个仓库或小团队，直接放在仓库里最简单。

优点：

- 就近维护
- 上下文和代码强绑定
- 迭代快

缺点：

- 数量多了会带来上下文和维护成本
- 不利于跨仓库复用

### 2. 大范围共享适合 marketplace

当 skill 面向团队或组织共享时，marketplace 更合适。它带来的不是“酷炫”，而是更清晰的分发机制：

- 可集中列出可安装 skill
- 可控制版本与更新
- 可按人群分发不同渠道

### 3. 版本策略要明确

官方文档强调两点：

- 版本号可以写在 `plugin.json` 或 `marketplace.json`
- 不要两边都写；如果都写，`plugin.json` 会静默覆盖 marketplace 配置

这说明发布策略必须统一，不然很容易出现“以为更新了，实际没生效”。

### 4. 用 release channel 做稳定/试验分流

文档建议用两个 marketplace 指向不同的 ref 或 SHA，比如：

- `stable`
- `latest`

这样可以把稳定用户和早期试用用户分开，先在小范围验证，再推广。

### 5. 发布前必须先验证

官方文档给了非常明确的校验路径：

- `claude plugin validate .`
- `/plugin validate .`

在本地验证通过后，再把 marketplace 加进去测试安装，再装一个测试插件进行验证。

这背后的方法论很重要：marketplace 也是工程产物，不应靠人工肉眼检查。

## 五、把三篇材料合并后，得到的实操方法

可以把 skill 开发收敛成下面这个流程。

### Step 1: 先判断应不应该做成 skill

检查五个问题：

- 这是高频任务吗
- 这个任务的边界是否清楚
- 是否存在稳定最佳实践
- 是否有真实失败案例可沉淀
- 脚本/模板/资源是否能明显提高成功率

如果这五条里多数答“否”，先别做 skill。

### Step 2: 先写触发边界，再写内容

不要一开始就填大段说明。先明确：

- 触发它的典型用户请求长什么样
- 哪些近似请求不该触发
- description 应该怎么写才不宽不窄

### Step 3: 设计目录结构，做渐进披露

推荐最小结构：

- `SKILL.md`
- `references/`
- `scripts/`
- `tests/` 或 eval 样本
- 可选的 `assets/`

原则是：主文档短，细节拆出去，确定性能力交给脚本。

### Step 4: 在 SKILL.md 里优先写高信号内容

优先级建议如下：

1. 触发条件
2. 任务边界
3. gotchas
4. 推荐工作流
5. 输出约束
6. 需要时再补背景知识

### Step 5: 给 skill 补最小验证集

至少包含：

- 应触发样本
- 不应触发样本
- 关键产出验收标准
- 至少一个真实失败回归用例

### Step 6: 发布前做一次工程化检查

检查项：

- description 是否准确
- 是否有 gotchas
- 是否把常识写太多
- 是否存在可脚本化但仍靠自然语言硬写的部分
- 是否能被 `validate` 工具检查
- 是否有版本策略

## 六、对当前仓库的直接启发

对于 `fintools_skills` 这类仓库，后续新建 skill 时最值得坚持的不是“多写几个 skill”，而是提高每个 skill 的密度和可验证性。

优先建议：

- 每个 skill 明确写 trigger 和 anti-trigger
- 每个 skill 都保留 gotchas 区块
- 可确定执行的逻辑尽量下沉到 `scripts/`
- 复杂说明拆入 `references/`，避免 `SKILL.md` 过胖
- 为重要 skill 补最小 eval/测试样本
- 如果未来有组织级复用需求，再考虑 marketplace 化和稳定/试验双通道发布

## 七、一个可复用的判断标准

如果一个 skill 同时满足下面四条，就值得长期维护：

- 能稳定命中某类任务
- 比裸模型明显更可靠
- 有脚本/模板/资源支撑，不只是一段提示词
- 有办法验证它是否退化

如果只满足“写了一份说明”，那它更像备注，不像 skill。
