# Backtests LLM Operations Manual

This document is the strict machine-operator guide for `backtests/`.

It is written for LLMs and automation, not for casual human reading.
If this document conflicts with informal habits or guessed workflows, follow this document.

## Mission

Use `backtests/` to manage remote-agent backtests safely.

Allowed goals:

- inspect rules, pools, stocks, and simulators
- create or update `remote_agent` rules
- list existing pools for an agent and bind a chosen pool by language instruction
- create pools
- add or remove stocks from pools
- bind or unbind pools from rules
- trigger rule execution
- create and run simulators
- inspect generated agent or simulator outputs

Disallowed goal:

- fabricating derived results by writing directly into runtime result tables

## Required Operating Order

When the user asks to run a trading agent, use this order:

1. Confirm the runtime and backend are available.
2. Confirm the target `agent_id`.
3. Check whether a matching `rule.agent_id` already exists.
4. If missing, create the rule first.
5. Check whether the rule already has an assigned pool.
6. If no pool exists, ask the user to choose CLI or UI pool assignment.
7. Only after pool setup is clear, trigger rule execution.
8. Only after agent outputs exist, create or run simulators.

Do not skip the rule existence check.
Do not silently invent a pool for the user.

## Operational Surfaces

Preferred surfaces, in order:

1. backend HTTP API
2. existing UI
3. project scripts already intended for initialization or startup

Avoid direct SQLite or MySQL writes unless the task is explicit low-level repair and the user clearly asked for it.

## Startup Checks

Before any mutation:

1. Verify backend health with `GET /health`.
2. If the backend is down, do not proceed with API mutations.
3. If the database is missing, initialize it through `backtests/backend/scripts/init_db.sh`.
4. Treat missing token or missing remote-agent prerequisites as blockers to execution, not reasons to forge data.

## Allowed Mutable Objects

You may create or update these through API or UI:

- `rule`
- `pool`
- `pool_stock`
- `rule_pool`
- `simulator_config`

You may create `simulator` records only through the simulator creation endpoint.

## Conditionally Mutable Objects

`simulator` may be deleted through API when the user wants to remove a run.

`rule` and `pool` may be deleted through API only when the user explicitly wants deletion.

Avoid destructive cleanup as part of normal usage.

## Derived Runtime Objects: Do Not Fabricate

Do not directly insert, update, or backfill the following tables by hand or with ad hoc SQL just to make the UI show results:

- `agent_trading`
- `simulator`
- `simulator_trading`
- `stock_rule_earn`
- `pool_rule_earn`

Reason:

- `agent_trading` is derived from agent execution and sync logic
- `simulator` should be created through simulator workflow, not invented as fake history
- `simulator_trading` must be generated by running the simulator
- earn tables are derived summaries and must not be treated as user-authored state

If the user asks to repair corrupted derived data, pause and make that explicit before touching the database directly.

## Rule Policy

Current target rule type:

- `remote_agent`

When creating a rule for a trading agent:

- `name` must be meaningful to the user
- `type` should be `remote_agent` for the current workflow
- `description` should describe the remote trading agent
- `info` should contain the agent access information expected by backtests
- `agent_id` should be populated when known and treated as unique

Before creating a new rule:

- query existing rules and check `agent_id`
- if the same `agent_id` already exists, reuse that rule instead of creating a duplicate

## Pool Policy

Pool creation is allowed.

Pool assignment rules:

- create a pool only when the user intends to manage a stock set
- do not auto-create placeholder pools without user approval
- add stocks to pools only through API/UI flows
- remove stocks from pools only through API/UI flows

Stock mutations must respect current backend validation:

- the pool must exist
- the stock must already exist in the `stock` table

Do not directly increment or decrement `pool.stocks`.
That count is maintained by backend logic.

## Simulator Policy

Simulator creation is allowed only after rule and signal state are ready enough for the user goal.

Do not:

- create fake simulator results without running the simulator
- write directly into `simulator_trading`
- manually edit `earning_info` to mimic a finished simulation

To obtain simulator outputs:

1. create simulator through API
2. run simulator through API
3. read simulator details and trading records through API

## Recommended API Workflow

Base URL:

- `http://127.0.0.1:8888/api/v1`

### Inspect rules

```bash
curl -sS 'http://127.0.0.1:8888/api/v1/get_rule/rule_list?rule_type=remote_agent'
```

### Create a rule

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_rule/rule_list' \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "Agent 105",
    "type": "remote_agent",
    "description": "Remote trading agent 105",
    "info": "http://example.com/api/v1/agents/105/a2a/",
    "agent_id": "105"
  }'
```

### Ensure a rule exists by `agent_id`

Use this when the workflow requires create-if-missing behavior.

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_rule/rule/ensure_remote_agent' \
  -H 'Content-Type: application/json' \
  -d '{
    "agent_id": "105",
    "name": "Agent 105",
    "description": "Remote trading agent 105",
    "info": "http://example.com/api/v1/agents/105/a2a/"
  }'
```

### Inspect pools

```bash
curl -sS 'http://127.0.0.1:8888/api/v1/get_pool/pool_list'
```

### List pools for an agent before assigning one

Use this when the user asks questions like:

- "都有哪些 pool"
- "列出 agent 105 能选的 pool"
- "先看看这个 agent 现在绑了哪些 pool"

```bash
curl -sS 'http://127.0.0.1:8888/api/v1/get_rule/rule/agent/105/pools'
```

### Create a pool

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_pool/pool_list' \
  -H 'Content-Type: application/json' \
  -d '{"name":"agent-105-core-pool"}'
```

### Add a stock to a pool

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_stock/stock_list' \
  -H 'Content-Type: application/json' \
  -d '{"code":"000001","pool_id":1}'
```

### Bind pools to a rule

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_rule/rule/1/pools' \
  -H 'Content-Type: application/json' \
  -d '{"pool_ids":[1]}'
```

### Bind a pool to an agent by language-resolved `agent_id`

Use this when the user says things like:

- "assign growth pool 给 105"
- "把 alpha pool 绑定到 agent 105"

By `pool_name`:

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_rule/rule/agent/105/assign_pool' \
  -H 'Content-Type: application/json' \
  -d '{"pool_name":"growth"}'
```

By `pool_id`:

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_rule/rule/agent/105/assign_pool' \
  -H 'Content-Type: application/json' \
  -d '{"pool_id":1}'
```

### Run all stocks for a rule

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_rule/rule/run/1'
```

If the rule has no assigned pool, this endpoint returns a failure payload with `needs_pool=true` so the caller can branch into pool assignment instead of pretending execution succeeded.

For SSE-based execution:

1. `POST /get_rule/rule/{rule_id}/start`
2. read `execution_id`
3. `GET /get_rule/rule/{rule_id}/stream?execution_id=...`

### Create a simulator

```bash
curl -sS -X POST 'http://127.0.0.1:8888/api/v1/get_simulator/simulator_list' \
  -H 'Content-Type: application/json' \
  -d '{
    "rule_id": 1,
    "start_date": "2025-01-01",
    "init_money": 10000
  }'
```

### Run a simulator

```bash
curl -sS -X PUT 'http://127.0.0.1:8888/api/v1/get_simulator/simulator/1/run'
```

## UI Escalation Rule

If the user explicitly asks to use the UI:

- start or direct them to the frontend and backend
- do not replace the UI task with hidden API-only changes unless the user agrees

If the user asks by command:

- stay in CLI/API mode
- do not require UI interaction unless it is the only safe path

## Safe Mutation Checklist

Before any create or update:

1. Read current state first.
2. Confirm whether the object already exists.
3. Mutate the smallest necessary object.
4. Re-read the affected object after mutation.
5. Only then proceed to downstream execution.

## Prohibited Shortcuts

Do not:

- write raw SQL into result tables for convenience
- fabricate agent success by inserting rows into `agent_trading`
- fabricate simulator completion by inserting rows into `simulator_trading`
- change `rule.id`, `pool.id`, or other primary keys manually
- bypass backend validation for pool membership
- leave undocumented disposable runtime outputs in the repository as if they were source artifacts

## Failure Handling

If a requested action fails:

1. verify backend health
2. call the exact backend endpoint directly
3. inspect backend traceback or error output
4. only after the backend contract is understood, inspect frontend behavior

This rule is mandatory when the user says a frontend action does nothing.
