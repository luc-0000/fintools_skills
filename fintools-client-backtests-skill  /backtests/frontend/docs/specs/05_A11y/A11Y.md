# Accessibility

## Overall Posture

- Accessibility baseline comes primarily from Ant Design components.
- Project-specific accessibility work is limited.

## Strengths

- Tabs, modals, buttons, selects, and tables are library-provided.
- Text labels exist for most controls.
- Operator flows are fairly linear and predictable.

## Risks

### Clickable anchors used as buttons

- Several actions use `<a onClick={...}>` rather than semantic buttons.

### Live log screens are not announced accessibly

- Streaming log updates have no `aria-live` region.
- Auto-scroll behavior may be disorienting.

### Raw HTML log rendering

- Simulator log tab uses `dangerouslySetInnerHTML`.
- No extra semantic wrapping or sanitization cues are visible in UI code.

### Color dependence

- Status communication partly relies on tag color and colored emphasis values.

### Desktop bias

- Wide data grids and fixed desktop spacing imply weaker support for narrow screens.

## Recommended Improvements

- Replace clickable anchors with semantic buttons or links
- Add live region semantics for run logs
- Audit contrast on dark custom surfaces
- Define responsive fallback patterns for wide tables
